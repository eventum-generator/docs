There has always been a gap in synthetic data work. Describing a dataset takes one sentence — *a week of Apache access logs with a traffic spike at noon, mostly 200s with a burst of 500s during the spike*. Producing it takes knowledge that has nothing to do with the case itself: which input plugin shapes time, how a template picks values, where samples live, what the config schema expects. The person who owns the scenario — a SOC engineer rehearsing a detection, a developer preparing a demo, a tester sizing a load profile — is rarely the person who enjoys closing that gap by hand.

Eventum 2.6 closes it differently: with the [MCP server](/docs/mcp), the case is now described naturally, in language — and the platform stays underneath as the controlled foundation that decides what actually runs.

## Why now

Two things matured at the same time. AI agents became competent at multi-step technical work: they read schemas, write files, react to errors. And the [Model Context Protocol](https://modelcontextprotocol.io) gave that competence a standard socket — one server, any agent: Claude Code, Cursor, Codex, and whatever arrives next.

What did not change is the failure mode. An agent working from memory produces configurations that *look* right. A YAML file that reads correctly but does not run — or worse, runs and quietly produces the wrong data — costs more time than writing it yourself, because now the review is the work. Plausibility is not correctness, and for synthetic data the difference is the whole point: a detection rehearsed against slightly-wrong logs is a rehearsal of the wrong thing.

## The split that makes it work

The MCP server is built around one decision: the agent describes and drafts; Eventum verifies and executes. Nothing the agent believes is taken on faith.

It starts from facts, not recollection — the server hands the agent the actual plugin list, the actual config schema of the installed version, the actual template API generated by introspection. It writes inside a boundary: generator files in your workspace, with traversal and symlink escapes rejected at the door. And then comes the load-bearing step — the agent must pass its draft through Eventum itself. Validation loads the config into the same pipeline a real run uses. Previews render real events with real timestamps. Errors come back exact, with file paths stripped and secret values redacted, so the loop is honest in both directions: the agent sees precisely what is wrong, and nothing about your machine leaks along the way.

The agent repeats that loop — draft, validate, preview, fix — until the generator loads cleanly and the sample looks like what you asked for. Then it shows you, before anything is saved or started.

## The artifact is still a generator

This is the part we consider the real step forward for generator development. The output of the conversation is not a pile of generated rows — it is a generator: a plain, versionable set of files that runs on the engine like any hand-written one. Commit it, review it, ship it to a server, schedule it. It produces data deterministically, at any volume, on any day, with no model in the loop and no per-token cost — the agent was scaffolding, not the building.

That is the deliberate division of labor: natural language owns the *intent*, the platform owns the *behavior*. Cases get described the way people think about them; reproducibility, performance, and operational control stay where they belong — in an engine that does the same thing every time.

It scales past authoring, too. Mounted into a running server over HTTP, the same protocol lets an agent operate the fleet: register the generator it just built, start it, read scrubbed logs when something misbehaves, and propose the fix — behind the server's authentication, with write access off by default.

## Honest edges

The boundaries are explicit because the risks are real. Generators execute code by design — templates and scripts are programs — so write access over HTTP stays disabled until you enable it, and turning it on means trusting the network and the agent with code execution on the host. Agents also vary: the loop guarantees the generator *runs* and shows you what it produces, while judging whether the data matches your intent remains your review. The platform draws the lines; you still hold the pen at the checkpoints.
