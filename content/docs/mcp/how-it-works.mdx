The MCP server gives an agent three kinds of capability: **tools** it calls, **resources** it reads for grounding, and **prompts** — reusable task templates it can invoke. Together they drive one loop: build a generator, check it against the real engine, look at the output, and fix it.

## The authoring loop

A generator in Eventum is not a single file — it's a **directory**: a [`generator.yml`](/docs/core/config/generator-yml) plus any files its plugins reference — templates and sample data for a `template` event plugin, say, or other files for other plugins, or nothing beyond the config. The agent works on that directory the same way you would.

<Mermaid chart={`graph LR
    DISCOVER["Discover<br/>plugins & schemas"]:::step --> WRITE["Write files<br/>config + referenced files"]:::step
    WRITE --> VALIDATE["Validate<br/>real engine"]:::step
    VALIDATE --> PREVIEW["Preview<br/>timestamps & events"]:::step
    PREVIEW --> FIX["Fix"]:::step
    FIX --> WRITE
    PREVIEW --> DONE["Save / run"]:::done
    classDef step fill:#8282ef,color:#fff,stroke:#6b6bd4,stroke-width:2px
    classDef done fill:#22c55e,color:#fff,stroke:#16a34a,stroke-width:2px`} />

1. **Discover.** The agent lists the available input, event, and output plugins and reads the exact schema of each one — so it knows every field, default, and constraint before writing anything. It can also read the [in-template API reference](/docs/mcp/tools#resources) — what the `template` plugin exposes inside templates — and a set of worked examples.
2. **Write.** It writes the `generator.yml` and any files its plugins reference — templates and sample data for a `template` event plugin, other files for other plugins — each by a path relative to the generator directory.
3. **Validate.** `validate_generator` loads the generator and initialises every plugin against the real engine. The agent gets the same field-level errors the engine would raise — not a guess.
4. **Preview.** `preview_timestamps` shows the shape of the schedule (how many events, when), and `preview_events` produces real example events through the configured event and output plugins. The agent sees the actual output.
5. **Fix and repeat** until validation passes and the preview looks right.

Because validation and preview run the real pipeline, everything — parameter and secret substitution, plugin initialisation, event production — behaves exactly as it will when the generator runs for real.

## Two transports, one definition

The same tools, resources, and prompts are served over two transports. You choose the one that fits how you work.

| | Local authoring (stdio) | Live management (HTTP) |
|---|---|---|
| How it runs | `eventum mcp` as a local process next to your agent | Mounted into a running [Eventum server](/docs/core/cli/eventum-run) |
| Best for | Building and previewing generators on your machine | Operating generators on a shared or remote instance |
| Authoring tools | Yes | Yes |
| Live management | — | Yes — list, status, start, stop, register, unregister, logs |
| Access control | Local file access | HTTP **basic auth**; read-only unless writes are enabled |

The live-management tools (starting, stopping, registering and unregistering generators, and reading their logs) exist **only** over the HTTP transport, because only a running server has generators to manage.

See [Connect your agent](/docs/mcp/connect) for setup and [Tools & resources](/docs/mcp/tools) for the full list.

## Safety

- **No model, no keys, no cost.** Eventum runs no language model and stores no provider credentials. Your agent supplies the model.
- **Writes are gated.** Over HTTP the server is **read-only by default**: the agent can discover, validate, and preview, but cannot change files or control generators until you enable writes (`server.mcp.allow_write`). Over stdio, writing is on by default because it operates on your local files — a `--read-only` flag turns it off.
- **Authentication.** The HTTP endpoint sits behind the same basic-auth credentials as the rest of the server.
- **No secrets leak to the agent.** Before Eventum sends a failed validation or preview error — or a generator's logs read over HTTP — it scrubs the text: absolute file paths are reduced to names, and resolved secret values (from [secrets](/docs/core/config/secrets)) are redacted. The agent sees what went wrong, not where your files live or what your secrets are. Structured errors are scrubbed exactly; logs are free-form text, so their scrub is best-effort, but absolute paths are always reduced. `list_secrets` returns secret **names** only, never values — and the agent cannot add, change, or read values; that stays with the [`eventum-keyring`](/docs/core/cli/eventum-keyring) CLI.
- **No network access from the server.** The MCP server itself never makes outbound calls. Example generators are bundled in the package; links to larger example sets are just URLs your agent can choose to fetch with its own tools.

<Callout type="warn">
Enabling writes over HTTP (`server.mcp.allow_write`) is more than file access: a connected agent can write a template and preview it, and **templates execute code on the host** (they can run subprocesses and arbitrary Python — by design). Treat an HTTP server with writes enabled as granting code execution to anyone holding the credentials, and turn it on only for a trusted network and a trusted agent. The defaults — HTTP off, writes off — avoid this entirely.
</Callout>

## What it does not do

- It does not replace [Eventum Studio](/docs/studio) — Studio is the visual editor; MCP is the programmatic, agent-facing interface.
- It does not generate data by itself — it builds and runs **generators**, which produce the data through the normal [pipeline](/docs/core/concepts/generator).
